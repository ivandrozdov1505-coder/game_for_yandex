# AGENT.md

## Цель
Ты работаешь над проектом как аккуратный самостоятельный разработчик.
Твоя задача — не просто написать код, а собрать цельный, рабочий, понятный и расширяемый проект.

---

## Основные правила работы

### 1. Сначала думай, потом кодь
Нельзя сразу прыгать в реализацию.
Сначала всегда:
1. понять задачу
2. определить MVP
3. спланировать архитектуру
4. только потом писать код

Если задача расплывчатая — не делай хаос. Выбирай самый простой и устойчивый вариант.

---

### 2. Делай MVP раньше полировки
Сначала нужен минимальный, но честно рабочий продукт.
Не надо:
- сначала делать красивые анимации
- сначала делать сложные эффекты
- сначала городить 100 экранов
- сначала тащить лишние библиотеки

Сначала:
- рабочий цикл игры
- рабочая логика
- рабочий UI
- рабочий проигрыш / результат
- рабочие сохранения, если они нужны

Потом уже улучшения.

---

### 3. Не усложняй стек без причины
Предпочитай:
- HTML
- CSS
- JavaScript

Используй только то, что реально нужно.
Не подключай тяжёлые библиотеки, если задачу можно решить нативно.
Не используй фреймворк ради модности.

---

### 4. Архитектура должна быть чистой
Разделяй:
- состояние игры
- игровую логику
- UI
- данные контента
- сохранения
- вспомогательные утилиты

Избегай:
- огромных файлов на всё
- хаоса из глобальных переменных
- копипасты
- смешивания логики и интерфейса в одной куче
- запутанных зависимостей между модулями

---

### 5. Код должен быть понятным
Пиши код так, чтобы его можно было спокойно читать и развивать.

Требования:
- понятные имена переменных и функций
- короткие осмысленные функции
- минимум магических чисел
- минимум скрытой логики
- комментарии только там, где без них реально хуже

Не засоряй проект бесполезными комментариями.

---

### 6. Мобильная адаптация обязательна
Любой интерфейс сразу делай с учётом телефона.

Проверяй:
- ничего не вылезает за экран
- кнопки удобны для тача
- текст читаем
- блоки не ломаются
- нет горизонтальной прокрутки
- важные действия доступны без ювелирного попадания пальцем

Если UI нормален только на десктопе — значит UI плохой.

---

### 7. Делай рабочий продукт, а не имитацию
Нельзя подменять механику заглушками, если задача требует рабочую систему.

Плохо:
- фальшивые кнопки
- неработающие режимы
- экраны без логики
- “временно” сломанные части
- декларативные обещания вместо реализации

Хорошо:
- пусть меньше, но реально работает

---

### 8. Не ломай то, что уже работает
Если вносишь изменения:
- не переписывай всё без причины
- сохраняй рабочую структуру
- не тащи регрессии
- не удаляй полезную функциональность случайно

Если переписывание нужно — оно должно быть осознанным и оправданным.

---

### 9. Версии обновляй везде
Если в проекте есть версия, обновляй её согласованно во всех местах:
- package.json
- manifest
- constants
- VERSION
- footer/about screen, если есть отображение версии

Не оставляй проект с разными версиями в разных файлах.

---

### 10. README обязателен
Если проект создаётся или заметно меняется — обновляй README.

README должен содержать:
- что это за проект
- как запускать
- как устроен проект
- ключевые механики
- ограничения
- идеи для будущих улучшений

---

## Приоритеты при разработке

Порядок приоритета:
1. Работоспособность
2. Понятная архитектура
3. Простота поддержки
4. Адаптивность
5. Аккуратный внешний вид
6. Дополнительная полировка

Если приходится выбирать между красивой хуйнёй и рабочей системой — выбирай рабочую систему.

---

## Работа с UI
UI должен быть:
- простой
- чистый
- понятный
- визуально цельный

Избегай:
- перегруженности
- слишком мелких элементов
- плохого контраста
- случайных цветов
- сломанной сетки
- текста, который не помещается

---

## Работа с логикой
Игровая логика должна быть:
- предсказуемой
- расширяемой
- отделённой от представления
- без дублирования

Если есть таблицы событий, конфиги баланса, режимы игры — выноси их в отдельные структуры данных, а не хардкодь в 20 местах.

---

## Работа с состоянием
Храни состояние централизованно.
Изменения состояния должны быть понятными и отслеживаемыми.
Избегай хаотичных изменений из разных кусков UI.

---

## Работа с контентом
Если в игре есть:
- события
- режимы
- описания
- параметры
- баланс

то оформляй их как данные, а не размазывай по коду.

---

## Что делать перед финалом
Перед завершением работы обязательно проверь:
- проект запускается
- нет очевидных ошибок
- все основные кнопки работают
- интерфейс не сломан на мобильной ширине
- README обновлён
- версии синхронизированы
- нет мусорных файлов и мёртвого кода

---

## Запрещено
- начинать с хаотичного кодинга без плана
- без причины тянуть тяжёлые зависимости
- смешивать всё в один файл
- оставлять сломанные элементы интерфейса
- делать нерабочие разделы “на будущее”
- плодить костыли там, где можно сделать нормально
- делать вид, что задача выполнена, если реализована только часть

---

## Нормальный подход
Сначала — структура.
Потом — ядро логики.
Потом — базовый интерфейс.
Потом — режимы и контент.
Потом — полировка.
Потом — README и финальная проверка.
